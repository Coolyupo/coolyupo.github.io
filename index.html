ch22 處理連鎖故障

連鎖故障舉例

  • 伺服器超載
    • 負載平衡端點存在(A&B)，單B端點故障時，導致A端點的QPS上升，進而導致A端點超載產生拒絕效應，連原先能乘載的QPS都無法達到。
  • 資源耗盡(在多個系統、程式共用硬體資源的情境下，部分服務的超載都可能會透過硬體資源直接影響到其他服務)
    • CPU
      • 請求上升
      • 佇列過長
      • 執行緒卡死
      • CPU死鎖
      • RPC Timeout
      • CPU 快取效率下降
    • 記憶體
      • 當機
      • 導致CPU使用率上升(GC的關係)
      • 快取命中率下降
    • 執行緒
      • 執行緒不足可能會影響到許多服務(以前遇過server上被zombie thread佔滿導致部分服務無法正常運作32767?)
    • 資源之間相依問題
      • 資源的耗盡有可能是其他資源出現問題所導致的(以前有遇過，CPU處理速度太慢導致Worker queue不斷堆疊，反而產生oom的情境)
  • 防止軟體伺服器超載
    • 進行負載壓力測試了解系統極限，測試超載失效模式
    • 提供降級的結果
      • 問題發生時，由個別服務決定提供品質較低(資源消耗量較少)的結果
    • 超載情況下主動拒絕請求
      • 軟體伺服器應該主動拒絕請求，與前面提到的adaptive throttling類似
    • 上層系統應主動拒絕請求，由上而下
      • 反向代理層拒絕
      • 負載平衡層拒絕
      • 個別作業拒絕
      • 個別服務(元件)容量規劃
  • 佇列管理
    • cpu使用，執行耗時
    • 佇列越長導致記憶體使用量越高
    • 當處理無法跟上佇列時應提早拒絕、進行轉移負載。
  • 丟棄負載 load shedding
    • 系統臨近超載時，主動丟棄一定量的負載
      • HTTP 503(Service unavailable)
      • FIFO 先入先出
      • LIFO 後入先出
      • CoDel 可控制延遲
  • 優雅降級 graceful degradation
    • 當系統開始丟棄負載時，服務可自行降低回應品質，減少運算量、運算時間。
    • 需要設計降級的決策指標(CPU、延遲、佇列長度、執行緒......)
    • 降級不該被經常觸發，平常也不會使用這個模式
      • 更顯示測試的重要性，確保該模式在真正發生的情況下能夠正常運作。
  • 重試
    • 遇到錯誤時進行重試
    • 過度的重試反而會造成連鎖故障
      • 被累加到後續QPS，造成伺服器過載，後端真正執行的工作反而很少
    • 設計重試應考量
      • 前面提到防止軟體伺服器重載
      • 隨機化、指數型遞增重試週期
      • 限制重試、避免無限重試
      • 試著將重試納入全域的預算分配(指定同一時間內系統可允許的重試次數)
      • 多角度檢視服務(ex:前端、後端、資料庫)，避免放大效應
      • 明確的錯誤返回代碼，能夠劃分不需要重試的永久性錯誤，與可重試的錯誤。
  • 解決連鎖故障的立即步驟
    • 增加資源
    • 停止健康檢查導致的作業當機(前面一值提到的sampling的副作用)
    • 重啟軟體伺服器(restart service or reboot? XD)
    • 丟棄流量
      • 這邊就類似Kubernetes readness probe，服務一起動湧入大量流量便馬上當機，永遠都修不好
    • 降級模式
    • 移除非關鍵Batch Job
    • 消除有害流量(會造成攻擊性或系統漏洞請求)
      • 比較多應該是無特別限制的大範圍搜尋(應該從程式設計就考量)

Ch23 管理關鍵狀態：利用分散式一致化來提高可靠性

  • CAP理論

    • Consistency — A read is guaranteed to return the most recent write for a given client
    • Availability — A non-failing node will return a reasonable response within a reasonable amount of time (no error or timeout).
    • Partition Tolerance — The system will continue to function when network partitions occur. > 當分散式系統發生問題時，往往無法滿足這三個要求。如Partition Tolerance(當網路分區發生問題時，至少一個或多個節點會被迫中斷，將不會是所有的節點都能夠被存取，資料當然也不會一致)

  • 領導者選舉與分散系統中的最終一致化(eventual consistency)

    • 這邊指就是像etcd，會經過定期的選舉來決定領頭者(主要的處理者)，透過傳遞使資料一致化
    • 資料庫的replication模式(master-slave架構)也是這種資料一致化的概念，只是沒有選舉的動作(沒有Availability)。
    • 如果不是一致化的架構，代表可能是用分散式的shard架構，則當叢集腦裂、損壞時可能會無法恢復，需要以另外的方式去做保險(就我所知很多使用shard機制的服務都會針對shard片段再做replication避免資料丟失無法還原)
    • 分散式一致化演算法
      • 當機不可恢復 crash-fail
      • 當機可恢復 crash-recover
        • 比較有用，很多時候可能會因為不知名外在因素(網路緩慢)導致重啟
      • 拜占庭式問題
        • 簡單來說就是有節點惡意發送不正確訊息時。
      • Paxos
        • proposer(提案者)
        • acceptor(接受者)
        • 當提案者收到總數一半+1人同意時(\(\frac{n}{2}\)+1)，提交提案並記錄
        • 不成熟，但是後續分散式演算法的啟蒙(raft.......)

  • 分散式一致化系統效能問題 > 普遍認為分散式一致化系統演算法速度太慢、資源要求過高、不適合搭建高處理能力且低延遲系統，未必正確，透過設計和系統優化、佈署策略都能夠有所改善。

  • Multi-Paxos

    • strong leader：在沒有發生錯誤、存在領頭者的情況下，提案者只需要對滿足法定人數的處理程序發送一次訊息，就可以保障系統達成一致化。
    • 原本的Paxos非strong leader，在Leader不常異動的情況下，原先Paxos階段一Prepare/Promise(提案者發送序號給接收者，接收者只能接收最高序號的提案)這件事情會變得單純，Leader持續提案、Acceptor持續接收(不會有發送舊的序號要驗證這件事情?)

  • Fast Paxos

    • Client直接向acceptor發送propose而非透過leader(傳統為Client透過Proposer、再由Proposer發送propose給acceptor)
    • 兩條訊息變成一條
    • 直覺上比較快，但還是得考慮網路延遲問題(若Client速度很慢，會由1個慢速網路(往proposer)變成N個慢速網路(往acceptors)反而拖垮效能)

  • 複本的數量

    • 採多數法定
      • 2f+1，f:失敗
      • 非拜占庭容忍
      • ex:3個節點，可允許1個失敗
    • 拜占庭容忍
      • 3f+1，f:間諜數量(有節點在惡搞)
      • 這邊好像在Block chain上應用比較多(公有鏈?)，需要積極去考量防偽

ch24 分散式任務排程系統

Cron

  • 最常見的排成格式(crontab)
  • 由單一元件組成crond，是一隻系統服務程式(daemon)，管理載入任務清單。
  • 從可靠性探討Cron
    • 任何的故障都可能會影響服務可靠性(單伺服器運作)
    • crond重啟需保留的只有組態檔(crontab)，啟動任務後不再追蹤狀態
    • anacron是特例
      • 會執行當機期間的任務
      • 但其實只對短周期任務有作用，每天只執行一次之類的
      • 對電腦OR工作站維護紀錄有幫助，做法就是將Cron執行的紀錄再做一次紀錄XD ### 大型系統的Cron
  • 這邊在講的其實就是Kubernetes中的Cronjob XD
  • 將實際處理序與實體機器分開，故障發生時能自動移轉(增加可靠性)
  • 追蹤兩個選項
    • 將資料儲存在可用度很高的外部分散式儲存裝置上
    • 系統內部自行儲存一些(很小量的)狀態資料，(Kubernetes之於etcd的作法)
  • 大型Cron系統的驚群效應(thundering herd)
    • 在大量Crontab任務的情況下，要小心處理將任務時間平均設置，避免大家都在同個時間點執行大量作業(也許是午夜)，反而造成系統超載問題。
    • 即便如此，Google的排成任務負載仍是尖銳。

ch26 資料完整性 : 讀寫一致

超高資料完整性的策略考量(本身有矛盾與些許互斥)

  • 線上時間
    • 通常以可用率(availability)，代表服務可被使用者使用的時間
  • 延遲
    • 服務對使用者回應時間
  • 規模
    • 服務的使用數量、維持服務水平最高負載
  • 創新速度
    • 合理成本下的服務創新跌代
  • 隱私
    • 這邊只探討刪除，使用者刪除資料後(軟刪除)，資料必須在合理時間內真正摧毀 >ACID >原子性（Atomicity） >一致性（Consistency） >隔離性（Isolation） >持久性（Durability） >BASE >Basically Available(基本可用） >Soft state(軟狀態） >Eventually consistent(最终一致性）

備份與封存

  • 封存(archives)
    • 將資料長時間安全儲存，滿足稽核、鑑識與法律遵循要求。(交易日誌、財務資料?......現實生活的話可能就是所得稅申報?)
  • 雲端環境下備份難點與需求
    • 難點
      • 使用混合交易與非交易(non-trasaction)型備份與復原，恢復的資料不一定是正確的
      • 不停機的情況下，有同時操作不同版本風險(Rolling Update?)
    • 雲端服務API需求
      • 資料區域性與快取
      • 區域合全域資料分布
      • 強一致性、最終一致性
      • 資料持久性、備份與災害復原
  • 備份就像納稅一樣
    • 需要持久付出的代價
    • 交付一套復原系統而非備份系統
    • 複製機制與冗餘不代表可恢復性
  • 保留期限
    • 資料備份的儲存時間也許要考量
    • 長期下來也是花空間跟花$，需要設計如何Rotate
  • 保障資料完整性手段
    • 軟刪除(似資料庫修改Delete欄位的方式，不在第一時間做刪除，用以處理誤刪或一定時間內快速復原)
      • 由API等應用操作的誤刪(非一般軟刪除)可能會再增加一層懶刪除(lazy deletion)，刪除當下服務馬上不可使用，但由雲端供應商多保留幾周再徹底銷毀。(之前在Azure Blob上好像有看到相關的內容)
    • 備份和相關的復原方法
      • 備份與還原的方法
      • 完整or增量備份的復原點頻率
      • 備份儲存位置
      • 備份保留時間
    • 複製機制
      • 儲存設備或服務架構具備複製的機制，Hardware RAID、MDADM、DB Replication......
      • 在資料存放到越大時難度將越來越難
    • 早期預警
      • 資料檢查與校驗動作
      • 監控
      • 確保恢復策略正常工作(演練?)
  • 心態
    • 保持初學者心態
      • 永遠會有自己不了解的BUG
    • 信任但還是要驗證
      • 即便在完美都是需要驗證!
    • 不要一廂情願
      • 系統總有一天會出錯
      • 人類天生不適合持續性，自動化很重要
      • 資料復原很重要、需時常演練
    • 縱深防禦
      • 任何完美的系統都有存在BUG的可能
      • 在允許的情況下多加幾層防禦能夠加速檢測問題的存在
      • (如同起司理論的意思XD)

  • 假設 - 排除手法
  • 從書本上抄寫的理想流程圖(用markdown flowchart.js重新寫了一遍樣式不同、順序一致)
  • 常見陷阱
    • 誤解(不當理解)系統故障的現象，浪費錯誤預算
    • 太快將問題歸納為極端因素
    • 試圖將問題解決歸納至與當前系統故障相關的一些問題，未意識到只是巧合
      • ex，程式故障同時系統資源指標微幅上升，試圖解決微幅上升指標而非程式故障問題 ### 實務
  • 故障報告
    • 實際現象
    • 預期原因
    • 如何重現
  • 問題分類
    • 盡可能讓系統恢復服務(需要捨棄一些東西)
    • 與尋找問題根源有互斥的感覺
  • 檢查與診斷
    • 這邊描述的就是像kubernetes中liveness probe功能類似
    • 簡化縮小問題範圍
      • 問題分解/Divide&Conquer
      • what(目前執行什麼)/why(系統為何如此操作)/where(資源被用在哪裡)
  • LOG
  • 測試與修復

使故障排除更容易

  • 增加可觀測性，結構化日誌
  • 好的系統設計帶你上天堂......
  • 在除錯的時候減少運氣與經驗的成分，讓新手也能有效解決問題，有助於縮短MTTR(平均修復時間)

    • 這邊就是monitoring與常說的aggregate概念雷同，當系統發生一件問題時可能會觸發多個警報(儘管造成的事件是同一個)。
    • 但過度的過濾也許會反而遺漏掉一些危險性警報(需注意)
  • 標籤化
    • 與現今的issue管理概念一樣，團隊訂製有意義的標籤，進行更明確、有效率的的區分，也有助於後續分析。
  • 分析
    • 透過事件追蹤與分析便能夠彙整與統計，進而做出一些結論或預測。

ch17 測試可靠性

傳統測試

  • 單元測試 unit test
    • 最小單位執行(通常軟體部分)
  • 整合測試 integration test
    • 將單元測試元件組合成較大的系統元件，透過執行整合測試驗證系統元件正確性
    • DI mock 取代真實資料庫進行測試
  • 系統測試 system test
    • 冒煙測試(smoke test) : 檢測非常簡單但非常重要的系統行為，又被稱健全測試，若無法通過代表不需繼續向下測試
    • 效能測試(performance test) : 通過smoke test代表正確性已受到確認，透過效能測試確認系統負載可接受的範圍(用在Resource Manager 設定cpu/memory request、limit這一塊應該很有用)
    • 回歸測試(regression test) ： 用以保證之前的Bug不會重現，將過去的bug修復測試整合進來，確實降低重蹈覆轍的風險。
  • 量產測試，又稱黑箱測試(Black-box testing)
    • 系統已上線，透過正式環境直接互動進行測試，用來評估可靠性。
  • 組態測試
    • 應該是類似於yamllint、dry-run這一類，針對設定檔案進行驗證，確保在自動化執行的過程不會發生意外
    • 也有可能是語意判斷類似現今Open Policy Agent這種模式
  • 壓力測試
    • 確認系統效能邊界，ex. 最大QPS、最大連線數量
  • 金絲雀測試
    • 逐漸讓使用者體驗新的版本或設定
    • 可以從階段性的錯誤發現定義問題等級
    • (我自己覺得這是一種UAT的展現，同時也適用在測試量能不足的情況下，但設計這個規則本身就蠻複雜的) >任何時候繞過測試的軟體功能，皆有可能造成嚴重後果，測試很重要 >MTBF：平均故障間隔 Mean Time Between Failure >MTTR: 平均故障修復 Mean Time To Repair >MTTF: 平均失效時間 Mean Time TO Failure

ch20 資料中心內部的負載平衡系統

  • HTTP Requeset/Client tasks/Resource Manager(CPU.......) ### 理想情況
  • 任一時間點，最忙OR最不忙永遠消耗同樣數量CPU(這代表工作能夠有效的被多工處理，不會有單一CPU占用過高的問題)，服務的負載均勻發送給後端作業。 ### 識別異常任務
  • 流量控制
    • ratelimit的做法
    • 通常僅能保護在極端超載的情況下，現實系統可能因各種因素提前就會超載了
    • 可透過調整參數限制，但終究還是有侷限性。
  • 跛腳鴨任務
    • 後端作業的狀態可能為
      • 健康
        • 初始化成功，正在處理請求
      • 拒絕連接
        • 無回應狀態(啟動或停止或異常中)
      • 跛腳鴨狀態
        • 可以接受服務請求，但明確要求用戶端停止方送請求
    • 停止程序通常為
      • 作業調度系統發送SIGTERM信號給作業
      • 作業進入跛腳鴨狀態，請求用戶端改送其他端點
      • 進入跛腳鴨狀態前已發送得請求會繼續處理
      • 隨著工作完成，逐漸釋放資源、活躍請求降低為0
      • 在設定的時間內未完成所有工作，再由作業調度系統進行強制關閉的動作 > 這一塊指的應該就是常見的graceful shutdown的概念 ### 子集劃分，選擇合適的子集
  • 隨機子集
    • 直覺、但容易負載不均衡
  • 確定性演算法
    • 簡單來說就是透過計算目前用戶端、子集數量，進而決定每一輪分配給各子集的用戶(連線)數量，確保再用戶量多的情控下該輪每個子集負載能夠落實均衡。

負載平衡原則

  • 簡單輪詢
    • 俗稱的Round Robin(輪流)
    • 容易有問題=>最高負載作業比最低負載作業都2被CPU
      • 子集過小時，單一子集負載量上升
      • 請求處理成本不同，但卻依舊是輪流分配
      • 伺服器本身具備差異
      • 其他因素
  • 最少負載輪詢
    • 兩個很重要的限制
      • 有效請求的數量不代表後端容量(可能請求數很低但Workload很重或是本身機器資源偏少)
      • 每個用戶端的有效請求不包括其他用戶端
        • 這邊指的可能有很多人(連線)因為該後端連線數較低而同時湧入，反而造成壅塞。
  • 加權輪詢
    • 簡易與最少負載輪詢的改進
    • 透過後端指標加總，進行能力值區分，進行輪詢的分配調整。

ch21 處理系統超載

QPS陷阱

  • 不同請求可能需要不同的資源來處理，若只按照QPS無法正確衡量。
  • 絕大部分情況下使用CPU數量作為資源分配主要依據即可(題外話，kubernetes的基本HPA也是這樣)
    • 在有GC(Garbage Collector)的環境裡，記憶體的壓力自然而然會變成cpu壓力，因為記憶體不足意味著要持續處理資源回收的問題。
    • 相較cpu，其餘的資源較容易透過設定而減少資源短缺的情況 ### 為每個使用者設定限額
  • 為每一個服務設定上限的cpu limit、使用者限額。
  • 以google的情境，可能設定的總cpu limit會大於總共的limit(超賣的問題)
    • 原因在於根本很少情境下能夠用滿cpu limit.......
    • 用在現今雲端提供商應該也有許多相似的情境(尤其Network超賣) ### 用戶端測的節流機制
  • 額度達到時(超載時)，迅速拒絕請求，返回額度不足的訊息
    • 單純回傳額度不足，可以減少一部分服務資源使用量，但即便如此在發送拒絕的過程也會造成資源的消耗，仍然有可能進入超載狀態
    • 採用流量調控技術(adaptive throttling)技術實現用戶端流量控制，當達到配額不足時，由用戶發起端自行限制速度，並由本機直接回應失敗，而不會發到網路層(我猜這邊應該是用到快取的方式)
    • 用戶端拒絕請求的機率： \[max(0,\frac{requests-K \times accepts}{requests+1})\]
      • requests: 應用程式發廚的所有請求總數和
      • accepts: 後端作業接受的請求數量
      • requests & accepts 一開始的總和應相同，隨著時間與負載上升，accepts將低於requests，當requests大於K倍的accepts，則代表開始機率性拒絕。
      • K可修改，降低使演算法激進，提升則緩和，一般情況推薦K=2 ### 重要性(請求重要性應該被標記)
  • CRITICAL_PLUS：非常重要，拒絕的話會產生嚴重可見問題。
  • CRITICAL： 拒絕會產生可見問題，但不到最重要的情況。
  • SHEDDABLE_PLUS：可容忍的錯誤、可等待(重試)的請求。
  • SHEDDABLE: 不太重要，偶爾完全不用的情況。

連線造成的負載 - 應透過調整參數、降低頻率、動態建立、銷毀連線等進行最佳化 批次處理作業的短暫連線超載，強制使用批次處理代理(batch proxy)，將超載需求轉發負載，用以保護現行系統真正的後端作業正常。

Ch03 擁抱風險

可用性公式:

  • \((基於時間)可用性=\frac{系統正常執行時間}{系統正常執行時間+停機時間}\)
  • \((整體)可用性=\frac{成功請求數}{請求總數}\)

客戶對服務的風險(錯誤)容忍度

  • 需求可用性水平
    • 使用者預期的結果&服務內容
    • 是否為商業服務(與收入直接關聯)
    • 競爭對手
    • 主要客戶族群
  • 錯誤的影響層面
    • 故障類型分類與影響層面、解決方法
  • 成本
    • 以google SRE為例，可用性的計算是以小數點後幾個9為依據
    • 考量如將可用性提升99.9%->99.99%時所需花費的成本與其改進後的預期收益是否取得平衡 ### 使用錯誤預算的動機
  • 錯誤的發生可能來自於下列
    • Software Fault Tolerance，軟體容錯程度脆弱(exception or error)
    • Test 不夠充足、完善的測試
    • Deploy Frequency 發布頻率過快，與上述兩點息息相關
    • Canary Deploy 金絲雀佈署與其規模，一種UAT的測試手法
  • 匡列錯誤處理預算
    • 定義自身系統產品SLO，確認服務預計正常執行的總時間
    • 第三方公正系統決定正常運行時間(uptime robot?)
    • 透過上方的時間可用性公式可計算出系統不可靠性預算
    • 目前運行的時間高於SLO也就代表還能更發布新的版本(還有錯誤容忍的預算在)
  • 當頻繁違反SLO時可考慮
    • 版本Roll Back
    • 降低發布頻率 > 100%可靠度基本上不切實際，有效面對風險與管理比較重要。

Ch04 服務水準目標

SLI (Service Level Indicator) 服務水準指標

為某套系統服務定義的定量指標，例如: - 請求延遲(request time) - 錯誤率(如 http 200 or 非200) - 系統處理能力(ex. QPS) - 可用性(服務可用時間百分比)，或稱服務產出(yield) - 「格式正確的請求處理成功比例」 - 持久性(資料能夠完整儲存的時間) 透過測量將數據收集計算，統計平均、百分比等等 ### SLO (Service Level Objective) 服務水準目標 為某套系統達成SLI目標值或目標範圍，ex: - \(SLI \leq 目標值\)
- \(範圍下限\leq SLI \leq 範圍上限\) - Average Requeset Latency < 100ms (100ms為SLI，意思為小於SLI即達成SLO) ### SLA (Service Level Agreement) 服務水準協議 服務與使用者之間明示或暗示性協議，未達到SLO產生的後果(退款或罰款或訴訟之類的) > 大多數討論「SLA」時，其實是在講SLO~

指標的實務運用

  • Application(user參與):
    • 系統效能、可用性、延遲
  • 儲存系統
    • 延遲、可用性、持久性
  • 大數據系統
    • 系統效能、資料輸入至轉出的時間
  • 正確性 > 收集指標(metrics)的方式 Borgmono or Prometheus，也需要定義好所謂的sampling rate(我想這邊跟open tracing概念相去不遠)

定義目標

  • 以系統全面性考量，而非以當前狀態
  • 保持簡單SLI彙整，可簡化整個過程
  • SLO越少越好
    • 選擇足夠象徵(涵蓋)系統的SLO，確保每一項SLO都是必要
  • 不要追求完美
    • 先透過鬆散的目標，再隨著時間驗證逐漸收斂標準 > SLI與SLO是進行維運決策時的重要元素(透過定量指標與違反與否決定相對應的處理行為) ### SLO會建議使用者預期
  • 保留安全區
    • 對內使用較高SLO、對外使用較低SLO，可做為問題緩衝處理
  • 實際的SLO不要過高
    • 實際的SLO比告知消費者的SLO還高，會導致使用者產生依賴
      • 應採取ratelimit、限速等策略，也能讓系統處在相對低負載，而非有多少給多少
